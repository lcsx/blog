# 1 数据库事务机制

# 1.1 undo和redo日志

数据的写入直接操作数据文件是非常危险的事情。如sql语句在执行过程中，系统重启，会影响存储过程，哪些记录修改与否不得而知。

mysql总共有5种日志，其中只有redo和undo日志与事务有关。

日志就是数据文件的副本

sql语句要操作什么数据，mysql先拷贝到undo日志中，增删改查等操作记录到redo日志中。

如果这些操作没什么问题，redo日志就会同步到数据库。即使有意外情况，如服务器死机、断电，当mysql重启后继续按redo日志执行同步即可。

如果不想继续执行同步，根据undo日志，将事务回滚，数据文件回复到事务之前的状态了。

# 1.2 事务机制

RDBMS(关系数据库管理系统) 其实就是 SQL语句 + 事务(ACID)

事务是一个或者多个SQL语句组成的整体，要么全部执行成功，要么全部执行失败。

使用场景：秒杀商品，但没支付。那么事务要回滚，而不能提交。

# 1.2.1 开启事务机制【START TRANSACTION】

【默认情况下】mysql执行 每条 SQL语句都会自动开启和提交事务

【多条SQL纳入到一个事务下】 要手动管理事务

START TRANSACTION
SQL 语句
[COMMIT | ROLLBACK]

START TRANSCATION;
DELETE FROM t_emp;
DELETE FROM t_dept;
SELECT * FROM t_emp;        
SELECT * FROM t_dept;
# 事务没有提交，修改都是在日志文件中，操作记录在redo日志中
# 所以select查询语句也是到redo日志中查询数据，结果是无数据
# 但是真实的数据表中是有数据的，打开navicat可以看到表中仍有数据

# 提交事务
COMMIT;
# 回滚事务
# ROLLBACK;

# 1.2.2 ACID

  1. 原子性：一个事务中的所有操作要么全部完成，要么全部失败。事务执行后，不允许停留在某个状态。
  2. 一致性：不管在任何给定的时间、并发事务有多少，事务必须保证运行结果的一致性。
  3. 隔离性：要求事务不受其他并发事务的影响，如同在给定的时间内，该事务是数据库唯一运行的事物。
  4. 持久性。事务一旦提交，结果便是永久性的。即使发生宕机，仍然可以依靠事务日志完成数据的持久化。

# 1.3 事务的隔离

序号 隔离级别 功能
1 read uncommitted 读取未提交数据
2 read committed 读取已提交数据
3 repeatable read 重复读取
3 serializable 序列化

没有通用的事务隔离，要根据业务实际设置。

# 1.3.1 买票【READ UNCOMMITTED】

  1. A事务看到1A坐席状态“未售出”，便update语句修改 记录的状态“已售出”，但没有提交事务；
  2. 由于事务隔离，B事务也看到1A坐席状态“未售出”，update语句修改 记录的状态“已售出”，并很快提交事务。此时数据文件发生改变了。
  3. 此时，引发A事务发生回滚事件。虽然没有引起数据错误，但是频繁的购买失败用户体验不好。
  4. 此场景下，应该允许当前事务可以读取其他事务的状态。B事务发现A事务购买了1A坐席，B事务可以购买其他坐席。

所以，👆用read uncommitted【读取未提交数据】

SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
# SET SESSION 设置当前会话的隔离级别  非全局的

navicat面板1

START TRANSCATION;
UPDATE t_emp 
SET sal=1;

navicat面板2

START TRANSCATION;
SELECT ename,sal FROM t_emp;

顺序执行，看出2中查询的数据是未修改的。

修改navicat面板2

SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT ename,sal FROM t_emp;
COMMIT;

此时，可以看到面板1的数据了

# 1.3.2 银行转账【READ COMMITTED】

事务A 向Scottz转账1000----> Scottz账户5000 <---- 事务b支出100

Scottz账户有5000，此时A事务转1000给scottz，但还没来及做任何操作时，事务B支出100。

如果A、B都正常commit，是不会有差错的。但当B事务有回滚操作时，就没有100的支出。如果允许A事务读取B事务的临时数据，那么结果是5900【少了100】。

SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
# sql...

# 1.3.3 用户下单了，但支付前商品涨价了【repeatable read】

事务A 下单 ----> 商品价格350 <---- 事务b 涨价

按照下单价格支付。

我们希望当前事务的执行不受其他事务影响。甚至其他事务的提交的结果也不影响当前事务。

【repeatable read】代表事务在执行中，反复读取数据，得到的结果是一致的，不会收到其他事务影响。

navicat面板1

# 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;  #1
# 开启事务  
START TRANSACTION;  #2
SELECT empno,ename,sal FROM t_temp;  #3
COMMIT;  #4

navicat面板2

START TRANSACTION;  #1
UPDATE t_emp  #2
SET sal=1;  #3
COMMIT;     #4

面板1、2随机交替执行，只要面板2中commi比面板1的select先执行，面板1 select查询出的结果就已被面板2的修改了，与预想的不一样。

因为undo日志记录，需要操作数据才会有，也就是 面板1需要一开始就执行select，然后即使面板2commit了，也不会影响面板1select查询的结果。

# 事务序列化

由于事务并发执行所带来的各种问题，前三种隔离级别只适用再某些业务场景中，但是序列化的隔离性，让事务逐一执行，就不会产生上述问题了。

事务序列化降低数据库并发性

一旦设置事务序列化，当前会话需要等待其他事务执行结束才会执行，不再出现并发事务。虽然可以避免业务场景问题，但会导致并发性下降。所以很少使用。

SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

# 2 数据导出与导入

# 2.1 数据导出与备份的区别

数据导出：导出的纯粹是业务数据 数据备份，备份的是数据文件、日志文件、索引文件等。【分：全量备份、增量备份（只备份变动的部分）】

# 2.2 SQL文件的导出与导入

数据导出可以分：

  1. SQL文档【数据量少时推荐】
  2. 文本文档【大】

# 2.2.1 导出SQL文件【mysqldump】

mysqldump 把业务数据导出成SQL文件，包括表结构

# mysqldump -uroot -p [no-data] 逻辑库>路径
mysqldump -uroot -p demo>D:/MBA/demo.sql

加上 no-data 只导出表结构。不加[no-data] 就是表结构跟业务数据都导出

# 2.2.2 导入SQL文件【source】

需要再mysql命令行中

# 先登陆数据库
mysql -uroot -p
# "-p"代表用密码登陆

sql > USE demo;
sql > SOURCE xx.sql;

# TXT文档的导入与导出【速度快】

需要先导出表结构，再导出txt文件

导入的时候，先导入表结构.sql文件，再导入txt文件

注意附加选项【第一个数据行】，并设置对应字段。

乱码

由于字符集原因，可能会有中文字符乱码，不过不影响导入
